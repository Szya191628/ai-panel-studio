# AI开发实习生远程作业：AI圆桌讨论Web App MVP

## 1. 背景与业务需求

### 1.1 产品定位

**AI Panel Studio** 是一款"AI圆桌讨论"本地Web应用：

- 用户输入任意讨论话题并指定参会专家人数（默认4人）
- 系统调用大模型动态生成一组主持人与专家嘉宾
- 展示每位嘉宾的姓名、职业、Title、立场与专属颜色标识
- 用户确认阵容后进入演播厅讨论模式
- 观看一场由AI驱动、实时推进的圆桌讨论

**核心价值主张：**
让任何人都能瞬间召集一支"虚拟智库"，围绕任意议题展开深度碰撞——主持人负责开场、追问、串联与收尾，专家之间自主"举手/抢答/补充/反驳"，讨论过程中持续生成共识与分歧，最终呈现一场沉浸式的AI圆桌实况。

### 1.2 目标用户画像

| 用户群体 | 特征描述 | 核心诉求 |
|---------|---------|---------|
| 知识工作者与研究人员 | 需要快速对一个议题进行多角度审视，时间有限 | 在几分钟内获得来自不同视角的深度分析与争辩摘要 |
| 产品/决策人员 | 在方案评审或战略讨论前希望预演各方立场 | 用AI圆桌发现盲点、检验假设，提升决策质量 |
| 教育与内容创作者 | 需要将复杂议题结构化呈现给受众，追求沉浸感与叙事性 | 获得一场可录制、可回放的"虚拟专家访谈"作为内容素材 |

### 1.3 用户痛点分析

1. **视角单一**：使用单一AI对话获得的分析往往立场单一，缺乏真实辩论中"对抗性思考"所带来的深度
2. **异步割裂**：多次单独询问不同角色的AI，上下文割裂，观点之间无法真正产生互动与碰撞
3. **过程不可见**：AI输出通常以最终结论呈现，缺乏"思考过程"和"观点演化"的实时感与沉浸感
4. **议题管理混乱**：用户希望同时探索多个议题，但不同讨论的上下文、进展与结论难以并行隔离管理

### 1.4 作业要求

本次作业要求必须使用 **Claude Code**（必须接入 **Deepseek V4 Pro** 模型），从0到1一人完成一个极简但有生产级质感的AI圆桌演播厅MVP。

#### 功能要求

| 功能模块 | 详细说明 |
|---------|---------|
| **首页** | 展示当前所有正在进行的讨论列表；用户可加入观察已有讨论，也可发起新讨论 |
| **嘉宾生成** | 用户输入话题与专家人数后，系统调用大模型动态生成主持人+专家阵容，展示姓名、职业/Title、立场、专属颜色标识，由用户确认后进入演播厅 |
| **演播厅模式** | 主持人负责开场、追问、串联和总结；专家根据当前transcript自主决定发言顺序（举手/抢答/补充/反驳），每次发言控制在1-2句；不允许机械式轮流发言 |
| **专家状态小窗** | 每位专家有独立小窗，实时显示Agent运行状态（待机/准备发言/发言中）、当前关注点或公开思考摘要；不展示真实的隐藏chain-of-thought |
| **实时共识与分歧** | 讨论过程中持续从发言中提炼共识与分歧，实时更新对应区域，不等到讨论结束后才生成 |
| **现场Transcript** | 显示发言人姓名与职业/Title，用与专家一致的色块区分；不显示"举手"等内部事件 |
| **总结** | 结束时主持人以自然语言进行总结，禁止将JSON原文显示到页面上 |

#### 技术要求

- ✅ 可本地运行的前后端分离项目
- ✅ 大模型 API Key 只能放在后端从环境变量读取，不得暴露到浏览器端
- ✅ 支持多个讨论并行：不同讨论的状态、事件流、transcript、共识与分歧必须互相隔离
- ✅ 实时更新使用 SSE、WebSocket 或等价机制
- ✅ 页面为中文UI，做响应式布局：不依赖整个页面滚动，各区域在自身容器内独立滚动；超宽屏、普通桌面与窄屏均需合理分配区域

## 2. 核心考察点

### 2.1 多范式融合的工程化架构（SDD+DDD+TDD）

考察候选人如何在不同阶段合理应用不同的开发范式，以有效管理AI上下文，避免大模型幻觉。

| 开发范式 | 全称 | 说明 |
|---------|------|------|
| **SDD** | Spec/Schema-Driven Development（契约/模型驱动） | 前期基于业务需求，驱动数据建模（讨论、嘉宾、发言、共识/分歧等实体）与API契约的精确定义 |
| **DDD** | Design-Driven Development（设计驱动） | 中期必须使用 UI UX Pro Max，以演播厅视觉风格与交互设计要素深度驱动前端组件与状态流转 |
| **TDD** | Test-Driven Development（测试驱动） | 后期针对核心逻辑（嘉宾生成、发言调度、共识提炼）严格遵循测试驱动流程，并包含完整的端到端（E2E）测试 |

### 2.2 其他考察点

- **必须使用 Superpowers 插件**：全程使用Claude Code作为唯一开发环境和核心工作流
- **AI协同效能**：考察对Prompt的结构化表达能力，以及如何引导AI解决具体工程问题
- **产品还原度**：对"AI圆桌演播厅"沉浸感与实时感的理解与功能落地，UI是否真的像在观看一场演播厅讨论
