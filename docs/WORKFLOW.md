# 开发过程思路 & 工作流说明

## 一、完整工作流：Claude Code + Deepseek V4 Pro 全流程开发

### 1.1 开发环境配置

- **主力工具**：Claude Code（接入 Deepseek V4 Pro 模型）
- **API 代理**：通过 `https://api.openai-proxy.org/v1` 调用 Deepseek V4 Pro
- **开发模式**：Claude Code 作为唯一开发环境，所有代码生成、测试运行、Git 操作均在其中完成

### 1.2 四阶段开发流程

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  SDD 阶段   │───►│  DDD 阶段   │───►│  TDD 阶段   │───►│  E2E 阶段   │
│  契约驱动    │    │  设计驱动    │    │  测试驱动    │    │  质量闭环    │
│             │    │             │    │             │    │             │
│ • 数据模型   │    │ • 视觉设计   │    │ • 测试用例   │    │ • 端到端测试  │
│ • Schema    │    │ • 页面布局   │    │ • 业务实现   │    │ • Bug修复    │
│ • API契约   │    │ • 交互逻辑   │    │ • Mock隔离   │    │ • 样例数据   │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
     ↓                   ↓                   ↓                   ↓
  Git Commit 1       Git Commit 2       Git Commit 3       Git Commit 4-8
```

**阶段一：SDD（契约/模型驱动）**
- 先定义 Pydantic 数据模型（models.py），确定所有字段和类型
- 再设计 SQLite Schema（schema.py），与模型严格对齐
- 最后封装数据库访问层（db.py），所有方法返回 dict
- 编写 30 个单元测试，全部通过后提交

**阶段二：DDD（设计驱动）**
- 定义 CSS 设计 Token（theme.css），建立统一视觉语言
- 以演播厅视觉风格驱动前端组件开发
- 先做布局（CSS Grid 三栏），再做组件（嘉宾卡片、发言气泡）
- 最后实现 JS 交互逻辑（SSE 连接、实时渲染）

**阶段三：TDD（测试驱动）**
- LLM 调用层使用 Mock 隔离，确保测试可重复
- 核心逻辑（嘉宾生成、发言调度）先写测试再实现
- 测试覆盖正常流程和异常路径

**阶段四：E2E（质量闭环）**
- 集成所有模块为完整应用
- 真实运行中发现并修复 Bug
- 添加样例数据和文档

### 1.3 每个模块的开发节奏

每个模块严格遵循 **Code Review → 测试 → Git Commit** 三步闭环：

1. **Code Review**：AI 生成代码后，人工审查逻辑和风格
2. **测试**：运行 `pytest tests/ -v`，确认全部通过
3. **Git Commit**：使用规范的提交信息（feat/fix/refactor 前缀）

## 二、AI 协同典型问题与解决路径

### 问题 1：函数定义遗漏（escapeHtml is not defined）

**现象**：前端点击"生成嘉宾"时报错 `escapeHtml is not defined`

**原因分析**：
- `escapeHtml` 函数在 `index.js` 中内联定义，但 `studio.js` 也需要使用
- 两个 JS 文件独立加载，函数不共享
- 开发时只测试了首页，未测试演播厅页面

**解决路径**：
1. 通过浏览器控制台定位错误发生在 `studio.js:88`
2. 检查 `utils.js` 发现该函数缺失
3. 将 `escapeHtml` 提取到公共的 `utils.js` 中
4. 提交修复：`fix(ui): 添加缺失的escapeHtml函数到utils.js`

**经验教训**：公共函数必须放在 `utils.js` 中，不能内联在页面 JS 里。

---

### 问题 2：API 请求无 Body 导致 500 错误

**现象**：点击"结束讨论"按钮弹窗提示 `Internal Server Error`

**原因分析**：
- 前端 `stopDiscussion()` 发送 POST 请求时没有 body
- 后端 `await request.json()` 对空 body 解析时抛出 `JSONDecodeError`
- FastAPI 默认不会捕获这个异常，导致 500

**解决路径**：
1. 使用 `curl` 直接测试端点，确认是后端问题
2. 检查 `app.py` 中的 `stop_discussion` 端点
3. 添加 `try-except` 包裹 JSON 解析，无 body 时 `conclusion` 默认为 `None`
4. 提交修复：`fix(api): 修复结束讨论接口无body时崩溃`

**经验教训**：API 端点必须防御性编程，假设所有输入都可能为空或格式错误。

---

### 问题 3：LLM 输出格式不稳定

**现象**：Deepseek V4 Pro 返回的 JSON 有时被包裹在 ` ```json ``` ` 代码块中

**原因分析**：
- LLM 输出格式不可预测，有时是纯 JSON，有时是 markdown 代码块
- 直接 `json.loads()` 会失败

**解决路径**：
1. 实现 `_extract_json()` 函数，兼容多种格式
2. 解析优先级：纯 JSON → code block → 正则提取第一个 JSON 对象
3. 添加完整的单元测试覆盖所有格式变体
4. 使用 `re.DOTALL` 标志处理跨行 JSON

**经验教训**：与 LLM 交互时，输出解析必须健壮，不能假设格式固定。

## 三、对「工程化 AI 开发」的理解

### 3.1 AI 是高效的代码生成器，但不是架构师

AI（如 Claude Code + Deepseek）可以快速生成大量高质量代码，但它不会主动思考：
- 这个函数应该放在哪个模块？
- 这个 API 的错误处理是否完整？
- 这个组件的复用性如何？

**工程化的核心**是人来定义架构和规范，AI 来执行实现。开发者的角色从"写代码"转变为"写 Prompt + 审查代码"。

### 3.2 分阶段递进是控制 AI 幻觉的关键

一次性让 AI 生成整个项目，会导致：
- 模块间接口不一致
- 遗漏边界条件
- 代码风格不统一

通过 SDD → DDD → TDD → E2E 的分阶段递进：
- 每个阶段有明确的产出物和验收标准
- 前一阶段的产出是后一阶段的输入
- 每个模块完成后立即测试和提交，问题不过夜

### 3.3 测试是人机协同的信任桥梁

AI 生成的代码可能有微妙的 Bug（如函数遗漏、异常未捕获）。测试用例是：
- 对 AI 输出的自动化验收
- 发现回归问题的早期预警
- 重构时的安全网

本项目 49 个测试用例全部通过，覆盖了 Schema、CRUD、LLM 调用、API 端点四个层次。

### 3.4 工程化 AI 开发的核心公式

```
工程化 AI 开发 = 清晰的架构设计
               + 结构化的 Prompt 引导
               + 严格的测试验证
               + 及时的 Bug 修复闭环
```

AI 负责"写"，人负责"想"和"验"。这三者的平衡，是工程化 AI 开发的核心。
